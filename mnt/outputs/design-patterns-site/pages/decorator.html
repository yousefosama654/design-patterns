The Decorator Pattern

Attach new responsibilities to an object dynamically without affecting other objects

The Starbuzz Coffee Problem

Imagine building a system for a coffee shop with different types of coffee (Espresso, House Blend) with add-ons (milk, soy, mocha, whipped cream). Each add-on has additional cost.

Bad vs Good Code Comparison

# Base class for beverages
class Beverage:
    def __init__(self):
        self.description = "Unknown Beverage"

    def get_description(self):
        return self.description

    def cost(self):
        raise NotImplementedError()

# Specific beverage types
class Espresso(Beverage):
    def __init__(self):
        super().__init__()
        self.description = "Espresso"

    def cost(self):
        return 1.99

class HouseBlend(Beverage):
    def __init__(self):
        super().__init__()
        self.description = "House Blend Coffee"

    def cost(self):
        return 0.89

# Subclasses with add-ons - PROBLEM!
class EspressoWithMilk(Espresso):
    def __init__(self):
        super().__init__()
        self.description += ", Milk"

    def cost(self):
        return super().cost() + 0.3

class HouseBlendWithSoyAndWhippedCream(HouseBlend):
    def __init__(self):
        super().__init__()
        self.description += ", Soy, Whipped Cream"

    def cost(self):
        return super().cost() + 0.5 + 0.7

# More combinations = More classes!
# Base class for beverages
class Beverage:
    def __init__(self, description="Unknown Beverage"):
        self.description = description

    def get_description(self):
        return self.description

    def cost(self):
        raise NotImplementedError()

# Concrete beverage classes
class Espresso(Beverage):
    def __init__(self):
        super().__init__("Espresso")

    def cost(self):
        return 1.99

class HouseBlend(Beverage):
    def __init__(self):
        super().__init__("House Blend Coffee")

    def cost(self):
        return 0.89

# Decorator base class
class AddOnDecorator(Beverage):
    def __init__(self, beverage):
        self.beverage = beverage

    def get_description(self):
        return self.beverage.get_description()

# Concrete decorator classes
class Milk(AddOnDecorator):
    def get_description(self):
        return f"{self.beverage.get_description()}, Milk"

    def cost(self):
        return self.beverage.cost() + 0.3

class Soy(AddOnDecorator):
    def get_description(self):
        return f"{self.beverage.get_description()}, Soy"

    def cost(self):
        return self.beverage.cost() + 0.5

class Mocha(AddOnDecorator):
    def get_description(self):
        return f"{self.beverage.get_description()}, Mocha"

    def cost(self):
        return self.beverage.cost() + 0.2

class WhippedCream(AddOnDecorator):
    def get_description(self):
        return f"{self.beverage.get_description()}, Whipped Cream"

    def cost(self):
        return self.beverage.cost() + 0.7

def main():
    # Espresso with Milk and Whipped Cream
    beverage = Espresso()
    beverage = Milk(beverage)
    beverage = WhippedCream(beverage)
    print(f"{beverage.get_description()} costs ${beverage.cost()}")

    # House Blend with Soy and Mocha
    beverage2 = HouseBlend()
    beverage2 = Soy(beverage2)
    beverage2 = Mocha(beverage2)
    print(f"{beverage2.get_description()} costs ${beverage2.cost()}")

Decorator Pattern Summary

Dynamic Behavior

Add behavior dynamically without affecting other objects. Each object remains independent.

Flexible Combinations

Combine multiple decorators in any order. One decorator class for each responsibility.

Real-World Uses

Coffee shop orders, UI components (borders, scrollbars), input stream compression and encryption.

Design Principles at Work

Encapsulate What Varies

Add-on costs and descriptions vary independently. Each decorator encapsulates one variation.

Favor Composition Over Inheritance

Instead of creating subclasses, we compose decorators. More flexible and maintainable.

Single Responsibility

Each decorator has one job: add one specific feature. Easy to test and modify.

Open/Closed Principle

Open for extension (new decorators), closed for modification (base classes stay stable).
