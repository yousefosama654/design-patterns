Behavioral Pattern

The Observer Pattern

Define a one-to-many dependency where when one object changes state, all its dependents are notified automatically

The Weather Station Problem

Imagine building a Weather Station app that monitors real-time weather data: temperature, humidity, and pressure. Initially, you have one display screen showing current conditions. But as your app grows, you need multiple displays:

  • Current Conditions Display
  • Statistics Display (avg, max, min)
  • Forecast Display

All displays must update immediately whenever weather changes. The system must be scalable—adding a new display shouldn't require changing the weather station code.

The Newspaper Subscription Analogy

The Observer pattern is like a newspaper subscription system. Let's break it down step by step:

Step 1: Publisher Goes Live

A newspaper publisher begins publishing daily editions, but has no one to send them to yet.

Step 2: You Subscribe

You subscribe to the publisher, and every new edition gets delivered to your doorstep automatically.

Step 3: You Unsubscribe

When you no longer want papers, you unsubscribe. Delivery stops immediately.

Step 4: Many Subscribers

People, hotels, and airlines constantly subscribe and unsubscribe. The publisher doesn't need to know who they are.

The Observer Pattern: The Publisher is the Subject, and Subscribers are the Observers. The Subject maintains a list of Observers and notifies them whenever something important happens.

Bad Code vs Good Code

Let's compare the direct approach (tightly coupled) with the Observer pattern (loosely coupled):

Problem: Tight Coupling 
class WeatherData:
    def __init__(self):
        self.temperature = 0.0
        self.humidity = 0.0
        self.pressure = 0.0

    def get_temperature(self):
        return self.temperature

    def get_humidity(self):
        return self.humidity

    def get_pressure(self):
        return self.pressure

    # Simulate updating weather data
    def set_measurements(self, temperature, humidity, pressure,
                         current_conditions_display, statistics_display,
                         forecast_display):
        self.temperature = temperature
        self.humidity = humidity
        self.pressure = pressure
        # Manually call display updates
        current_conditions_display.update()
        statistics_display.update()
        forecast_display.update()

class CurrentConditionsDisplay:
    def __init__(self, weather_data):
        self.weather_data = weather_data

    def update(self):
        temperature = self.weather_data.get_temperature()
        humidity = self.weather_data.get_humidity()
        print(f"Current: {temperature}°C, {humidity}%")

class StatisticsDisplay:
    def __init__(self, weather_data):
        self.weather_data = weather_data
        self.temp_readings = []

    def update(self):
        temperature = self.weather_data.get_temperature()
        self.temp_readings.append(temperature)
        avg_temp = sum(self.temp_readings) / len(self.temp_readings)
        print(f"Average: {avg_temp}°C")

class ForecastDisplay:
    def __init__(self, weather_data):
        self.weather_data = weather_data
        self.last_pressure = self.weather_data.get_pressure()

    def update(self):
        current_pressure = self.weather_data.get_pressure()
        forecast = "Improving" if current_pressure > self.last_pressure else "Cooler"
        print(f"Forecast: {forecast}")
        self.last_pressure = current_pressure

# Usage
weather_data = WeatherData()
current_conditions_display = CurrentConditionsDisplay(weather_data)
statistics_display = StatisticsDisplay(weather_data)
forecast_display = ForecastDisplay(weather_data)

weather_data.set_measurements(22.0, 65, 1013,
                              current_conditions_display,
                              statistics_display,
                              forecast_display)
weather_data.set_measurements(25.0, 70, 1012,
                              current_conditions_display,
                              statistics_display,
                              forecast_display)
⚠️ Problems:
  • set_measurements() must know about every display object
  • Adding a new display means changing WeatherData code
  • Tight coupling: WeatherData depends on concrete display classes
  • Not scalable: every change cascades through the code
  • Hard to remove displays without breaking things
Solution: Observer Pattern 
# Subject interface
class Subject:
    def register_observer(self, observer):
        raise NotImplementedError

    def remove_observer(self, observer):
        raise NotImplementedError

    def notify_observers(self):
        raise NotImplementedError

# Observer interface
class Observer:
    def update(self, temperature, humidity, pressure):
        raise NotImplementedError

# WeatherData class implementing Subject
class WeatherData(Subject):
    def __init__(self):
        self.observers = []  # list of all observers
        self.temperature = 0.0
        self.humidity = 0.0
        self.pressure = 0.0

    def register_observer(self, observer):
        self.observers.append(observer)

    def remove_observer(self, observer):
        self.observers.remove(observer)

    def notify_observers(self):
        for observer in self.observers:
            observer.update(self.temperature, self.humidity,
                          self.pressure)

    def set_measurements(self, temperature, humidity, pressure):
        self.temperature = temperature
        self.humidity = humidity
        self.pressure = pressure
        self.notify_observers()  # notify all observers

# Display classes implementing Observer
class CurrentConditionsDisplay(Observer):
    def update(self, temperature, humidity, pressure):
        print(f"Current: {temperature}°C, {humidity}%")

class StatisticsDisplay(Observer):
    def __init__(self):
        self.temp_readings = []

    def update(self, temperature, humidity, pressure):
        self.temp_readings.append(temperature)
        avg_temp = sum(self.temp_readings) / len(self.temp_readings)
        print(f"Average: {avg_temp}°C")

class ForecastDisplay(Observer):
    def __init__(self):
        self.last_pressure = None

    def update(self, temperature, humidity, pressure):
        if self.last_pressure:
            forecast = "Improving" if pressure > self.last_pressure else "Cooler"
        else:
            forecast = "Cooler, rainy weather"
        print(f"Forecast: {forecast}")
        self.last_pressure = pressure

# Usage - Much simpler!
weather_data = WeatherData()
current_conditions_display = CurrentConditionsDisplay()
statistics_display = StatisticsDisplay()
forecast_display = ForecastDisplay()

weather_data.register_observer(current_conditions_display)
weather_data.register_observer(statistics_display)
weather_data.register_observer(forecast_display)

# Just update the data - observers are notified automatically!
weather_data.set_measurements(22.0, 65, 1013)
weather_data.set_measurements(25.0, 70, 1012)
✓ Improvements:
  • Subject knows nothing about concrete observer types
  • Add/remove displays by calling register_observer() or remove_observer()
  • Loose coupling: Subject depends on Observer interface only
  • Highly scalable: new displays don't require changing existing code
  • Each observer manages itself independently
  • Follows SOLID principles (Open/Closed, Dependency Inversion)

Observer Pattern Summary

Core Concept

  • Subject maintains list of Observers
  • Notifies all Observers when state changes
  • Observers react to notifications
  • One-to-many relationship

Real-World Use Cases

  • Weather stations
  • Stock price tickers
  • Event listeners (GUI buttons)
  • Push notifications
  • MVC frameworks

Design Principles

  • Encapsulate What Varies
  • Program to Interface
  • Loose Coupling
  • Open/Closed Principle
  • Dependency Inversion

Comparison

  • Bad: Direct calls
  • Bad: Tight coupling
  • Good: Interfaces
  • Good: Loose coupling
