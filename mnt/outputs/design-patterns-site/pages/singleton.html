The Chocolate Factory Problem

Imagine a chocolate factory with a boiler to melt chocolate and milk. Only ONE boiler should ever exist. Let's see what goes wrong without Singleton.

🏭

Step 1 — The Factory Setup

The chocolate factory has one boiler to melt chocolate and milk. It starts empty, gets filled, boiled, then drained.

ChocolateBoiler

⚠️

Step 2 — The Problem

Without Singleton, the code allows creating multiple instances. Two separate boilers now exist — each with its own independent state.

Step 3 — Disaster Strikes!

boiler1 was filled and boiled. But boiler2 thinks it's still empty — so it gets filled again. Both boilers now contain chocolate, causing a production error.

❌ boiler1 state: filled=True, boiled=True
❌ boiler2 state: filled=True, boiled=False
💥 CRITICAL: Two instances — conflicting states!
Result: Wasted chocolate, production halted.
🔒

Step 4 — Singleton Saves the Day

With Singleton, get_instance() always returns the same object. No matter how many times you call it, there is only ever ONE boiler.

Bad Code vs Good Code

class ChocolateBoiler:
    def __init__(self):
        self.empty = True
        self.boiled = False

    def fill(self):
        if self.empty:
            self.empty = False
            self.boiled = False
            print("Boiler filled with chocolate and milk.")

    def drain(self):
        if not self.empty and self.boiled:
            self.empty = True
            print("Boiler drained.")

    def boil(self):
        if not self.empty and not self.boiled:
            self.boiled = True
            print("Boiler contents boiled.")

# PROBLEM: Creates two separate instances!
boiler1 = ChocolateBoiler()
boiler2 = ChocolateBoiler()

boiler1.fill()
boiler1.boil()
boiler2.drain()  # Won't work properly!
class BoilerOperations:
    def __init__(self):
        self.empty = True
        self.boiled = False

    def fill(self):
        if self.empty:
            self.empty = False
            self.boiled = False
            print("Boiler filled with chocolate and milk.")

class ChocolateBoiler:
    _instance = None

    def __init__(self):
        self.operations = BoilerOperations()

    @classmethod
    def get_instance(cls):
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance

# SOLUTION: Both get the SAME instance!
boiler1 = ChocolateBoiler.get_instance()
boiler2 = ChocolateBoiler.get_instance()

print(boiler1 is boiler2)  # True!

boiler1.operations.fill()
boiler1.operations.boil()
class President {
    private static President instance;

    private President() {
            // Private constructor prevents multiple instances
        }

    public static President getInstance() {
            if (instance == null) {
                instance = new President();
            }
            return instance;
        }

    public void speak() {
        System.out.println("I am the only President!");
    }
}

public class Main {
    public static void main(String[] args) {
        President p1 = President.getInstance();
        President p2 = President.getInstance();

        p1.speak();

        System.out.println(p1 == p2);
        // Output: true (same instance)
    }
}

How Singleton Works

Key Characteristics

Only One Instance

The class ensures exactly one instance exists throughout the application's lifetime.

Global Access Point

You access the instance through a static method like get_instance() from anywhere.

Lazy Instantiation

The instance is created only when first requested, not when the class is defined.

Common Use Cases

Database connections, loggers, configuration managers, thread pools, caches.

Global Variables vs Singleton

Global Variable

  • Always instantiated at startup
  • Can be modified from anywhere
  • Can't handle inheritance
  • No encapsulation
  • Hard to test

Singleton Pattern

  • Created only when needed (lazy)
  • Controlled access through methods
  • Can extend and override behavior
  • Encapsulated and protected
  • Easier to test and mock

Design Principles

Encapsulate What Varies

The singleton pattern encapsulates the instance creation logic and ensures it stays consistent. The "variation" (multiple vs single instance) is hidden inside the class.

S in SOLID: Single Responsibility

Your class should have one reason to change. A ChocolateBoiler shouldn't worry about being instantiated multiple times—the Singleton handles that.
